# 群唤醒增强插件 - 更新文档（v8.4.0）

## 更新日志

### v8.4.0 (2025-04-09)

# 更新后请点击插件配置然后保存一次配置，避免无新配置项引发错误
（我们不会使用.get方式回退默认值，以防覆盖你的配置却无任何提示，有错误请联系）

#### ✨ 新增功能

1. **灵活活跃记录时机**  
   新增配置项 `活跃方式`，支持三种模式：
   - `唤醒时`（默认）：在用户通过艾特、引用、昵称、概率等方式成功唤醒机器人时，立即记录群活跃时间和用户冷却。**（原行为）**
   - `llm请求后`：仅在机器人实际调用 LLM 并生成回复后，才记录活跃和冷却。适合需要“真正响应才消耗冷却”的场景。
   - `发送消息后`：在机器人完整发送完消息后（支持分段回复插件），才记录活跃和冷却。最贴近实际输出体验。
   - 实测框架只有在全部消息发送完成后才触发，不知道不同版本是否有变化，不过不会重复触发

2. **事件驱动活跃管理**  
   插件现在通过 `@filter.on_llm_response` 和 `@filter.after_message_sent` 钩子，精准捕获 LLM 响应完成和消息发送完成时机，避免因早期唤醒但被后续拦截导致的无效冷却。

#### 🔧 优化改进

1. **指令拦截逻辑优化**  
   - 管理员输入不存在的指令时，若规则开启了 `禁前唤醒`，同样会拦截该消息（之前版本会放行）。
   - 统一通过 `获取当前群规()` 方法获取规则，代码更清晰。

2. **强力拦截行为调整**  
   - `终止事件传播` 现在仅当规则中 `强力拦截` 为 `true` 时，才会设置 `群唤醒拦截` 标记并彻底阻止 LLM 请求。否则仅停止事件传播但保留 LLM 可能的后续唤醒（更温和）。

3. **系统指令拦截增强**  
   - 在 `llm请求前` 拦截器中增加了对 `禁用系统指令` 的检查，直接拦截 `/reset`、`/op` 等系统指令，不经过 LLM。

#### 🐛 问题修复

- 修复了在活跃期内，艾特/引用唤醒后仍可能被活跃间隔误拦截的问题。
- 修复了多个群共用一条规则时，`/设置群规` 显示“还有群xxx也使用了此规则”的文案错误。
- 修复了 `/群规则` 命令中时间类配置显示异常（函数调用缺失参数）。
